# Bias-Audit Framework

## Purpose

The bias audit checks whether the run overproduced agreement, clarity, or confidence.

It does not prove the output is true. It helps identify where the output may be misleading.

## Audit dimensions

### Founder-favoring bias

Does the output flatter the idea or smooth over hard objections?

### Prompt-leading bias

Does the wording imply that the product is already useful, wanted, or differentiated?

### Segment-collapse bias

Do different segments sound too similar?

### Synthetic-politeness bias

Do reactions avoid blunt rejection even when rejection would be plausible?

### Evidence-inflation bias

Does the synthesis convert reactions into evidence, demand, or validation?

### Control-failure bias

Did poor-fit or non-market-control perspectives react too positively?

## Required audit output

A bias audit should include:

- Strongest suspected bias
- Evidence from the run artifact
- Severity
- Required correction
- Whether the run should be revised before use

## Severity scale

| Severity | Meaning |
|---|---|
| Low | Worth noting but not run-breaking |
| Medium | Interpret with caution; revise synthesis |
| High | Run should be revised before being used |
| Critical | Run is misleading and should not be used |
