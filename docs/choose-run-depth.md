# Choose Run Depth
Signal Check is useful when the decision is important enough to deserve structure.
Not every question needs a full run.
Use this guide to choose between a normal chat answer, a Quick Run, a Full Run, and real-world testing.
## Decision matrix
| Situation | Use | Why |
| --- | --- | --- |
| Small wording question | Chat answer | The decision is reversible and does not need a full evidence trail. |
| Small feature detail | Chat answer or Quick Run | Use Quick Run only if the feature changes positioning, trust, or user expectation. |
| Homepage message | Quick Run or Full Run | Messaging can create category confusion and should be checked against segments and controls. |
| Product positioning | Full Run | The risk of self-confirming language is high. |
| Target segment choice | Full Run | Segment assumptions need fitting, adjacent, skeptical, poor-fit, and non-market-control views. |
| Feature prioritization before building | Full Run | The method helps avoid building from founder preference alone. |
| Beta recruiting message | Full Run | It needs claim boundaries, concrete audience assumptions, and real next-step planning. |
| Pricing, willingness to pay, conversion, or demand | Real-world test required | Signal Check cannot establish behavior, demand, or willingness to pay. |
| Safety, compliance, legal, medical, financial, or regulated claims | Real-world expertise required | Signal Check cannot establish safety, compliance, or legal sufficiency. |
## Chat is enough when
Use a normal chat answer when:
- the decision is small
- the change is reversible
- no claim boundary is at stake
- no meaningful segment difference is expected
- no public positioning or build decision depends on the answer
Examples:
- "Make this sentence clearer."
- "Suggest three button labels."
- "Tighten this paragraph."
- "List possible objections informally."
## Use Quick Run when
Use Quick Run when:
- the question matters but should not become a heavy project
- there are two or three plausible directions
- negative controls could reveal vague or flattering language
- the next action should still be a real-world check
Examples:
- homepage hero direction
- beta invite wording
- feature naming
- landing-page section order
- open-source README positioning
## Use Full Run when
Use Full Run when:
- the result may change product direction
- the target segment is uncertain
- the message could overclaim
- the product could be framed too broadly
- implementation work may follow immediately
- several stakeholders or future reviewers need traceability
Examples:
- choose a target audience
- decide whether to build a feature
- choose product positioning
- compare product categories
- prepare a campaign or landing page before real outreach
## Real-world test required
Move outside Signal Check when the question depends on real behavior.
Signal Check cannot answer:
- Will people buy?
- Will people pay this price?
- Will this convert?
- Will this retain?
- Will users trust this in a real context?
- Is this legally or medically safe?
- Is this market representative?
For these questions, use Signal Check only to design the next real-world test.
